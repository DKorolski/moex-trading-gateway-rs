# Broker contract

This document defines the initial broker-neutral contract. The contract is intentionally stricter than any one broker API.

## Identity

- `StrategyRequestId`: internal UUID for a runtime command.
- `BrokerAccountId` / `AccountId`: broker account id.
- `InstrumentId`: normalized symbol plus optional venue-native symbol.
- `InternalSymbol`: strategy-facing symbol such as `INTERNAL_TEST_FUT` in tests
  or a configured production strategy symbol in deployment.
- `BrokerSymbol`: venue-facing symbol such as FINAM `ticker@mic`.
- `ClientOrderId`: idempotency/correlation id generated by our system. For FINAM it must be non-empty and at most 20 characters.
- `BrokerOrderId` / `OrderId`: broker-assigned string id.
- `BrokerTradeId` / `TradeId`: broker-assigned trade/fill id.
- `RedactedValueFingerprint`: length and SHA-256 fingerprint for broker-native
  values that must not be stored raw but are still useful for diagnostics.

The id model is deliberately three-part:

```text
StrategyRequestId  -> internal command identity
ClientOrderId      -> broker-visible idempotency/correlation key
BrokerOrderId      -> broker-assigned order identity
```

`ClientOrderId` is not a UUID string. The full UUID is too long for FINAM. The mapping between all three ids must be durable before or at order emission.

Historical/read-only broker records may contain broker-native or manual
`client_order_id` values that do not satisfy our outgoing `ClientOrderId`
contract. In that case the normalized `client_order_id` remains empty and a
redacted fingerprint preserves the correlation fact without storing the raw
broker-native value.

Broker-native order comments are treated the same way for broker-neutral
snapshot streams. Raw comments can contain operator, broker, or account context,
so `Order.comment` must not be published in `OrderSnapshot` payloads. If a
comment is present in a broker response, the neutral order may keep only a
`comment_fingerprint` containing length and SHA-256. Raw comment capture, if it
is ever needed, belongs to a local-only broker-truth/debug store with an
explicit export policy, not to runtime-facing streams.

## Event classes

- `Position`
- `PortfolioSnapshot`
- `Order`
- `Trade`
- `MarketDataEvent`
- `BrokerReadiness`

Each event should preserve:

- broker/source timestamp where available;
- gateway receive timestamp;
- broker-native id where available;
- normalized symbol and optional venue-native symbol.

Market-data events also carry `source_kind` so downstream consumers can
distinguish historical/read-only polling, future live streams, and recovery
replay without relying on stream names.

## Readiness contract

Order-emitting systems can trade only when:

- auth is valid;
- account state is available;
- current positions are loaded;
- current orders are loaded;
- current trades or own-trade stream is ready;
- instrument mapping, tick/lot params, expiration, and schedule are validated;
- required market data subscriptions are live;
- first live bar is received;
- broker maintenance window and market schedule permit entry;
- rate limiter is active;
- clock skew is acceptable;
- operator pause is not active.

If readiness is not available at a model signal, runtime should wait for a bounded window rather than recomputing and replacing the decision silently. After timeout, it must record an explicit missed intent.

## Subscription contract

Subscriptions are normalized as intents:

- bars;
- quotes;
- order book;
- latest trades;
- own orders;
- own trades;
- positions.

Subscription state has phases:

- requested;
- connecting;
- syncing history;
- live;
- degraded;
- stopped.

## Order lifecycle

The adapter must tolerate:

- ack before order snapshot;
- trade before order snapshot;
- partial fills;
- repeated snapshots;
- reconnect replay;
- broker-side terminal state before our cancel request arrives.

ACK is not a fill. `CommandAck::Accepted` only means that the command path accepted or submitted the request. Final order state and fills come from order/trade events and reconciliation.

Phase 1 supports only MARKET/LIMIT/CANCEL. Stop, SLTP, and bracket semantics are disabled until separate FINAM contract tests prove their lifecycle.

The Phase 1 command enum intentionally exposes no SLTP command. FINAM SLTP stays
an API-research topic until a dedicated lifecycle contract and rejection tests
exist.

The M3 order path must follow `docs/m2-to-m3-readiness-gate.md` and
`docs/m3-order-path-design.md`: durable id mapping before network submit,
operator arming before endpoint use, and no blind retry after ambiguous
place-order timeout.

M3a starts this contract in code through `broker-core::order_path`. That module
is intentionally non-network: it defines state transitions, duplicate-id store
behavior, outgoing comment policy, operator arming, place/cancel preflight,
local durable restart/replay tests, and synthetic ACK construction without
calling any broker endpoint.
